= Conditions Evaluation Report (conditions)

The `conditions` endpoint provides information about the evaluation of conditions on configuration and auto-configuration classes.

== Retrieving the Report

To retrieve the report, make a `GET` request to `/actuator/conditions`, as shown in the following curl-based example:

[source,bash]
----
curl 'http://localhost:8080/actuator/conditions' -i -X GET
----

=== Response Structure

The response contains details of the applicationâ€™s condition evaluation.
The following table describes the structure of the response:

[%autowidth]
|===
|Path|Type|Description
|`contexts`|`Object`|Application contexts keyed by id.
|`pass:[contexts.*.positiveMatches]`|`Object`|Classes and methods with conditions that were matched.
|`pass:[contexts.*.positiveMatches.*.[\].condition]`|`String`|Name of the condition.
|`pass:[contexts.*.positiveMatches.*.[\].message]`|`String`|Details of why the condition was matched.
|`contexts.*.negativeMatches`|`Object`|Classes and methods with conditions that were not matched.
|`pass:[contexts.*.negativeMatches.*.notMatched]`|`Array`|Conditions that were matched.
|`pass:[contexts.*.negativeMatches.*.notMatched.[\].condition]`|`String`|Name of the condition.
|`pass:[contexts.*.negativeMatches.*.notMatched.[\].message]`|`String`|Details of why the condition was not matched.
|`pass:[contexts.*.negativeMatches.*.matched]`|`Array`|Conditions that were matched.
|`pass:[contexts.*.negativeMatches.*.matched.[\].condition]`|`String`|Name of the condition.
|`pass:[contexts.*.negativeMatches.*.matched.[\].message]`|`String`|Details of why the condition was matched.
|`contexts.*.unconditionalClasses`|`Array`|Names of unconditional auto-configuration classes if any.
|`contexts.*.parentId`|`String`|Id of the parent application context, if any.
|===
